Metodología Frontend: capacidades, rutas y validación

Cómo está construido el frontend de las capacidades del Grupo 5 (Activos e inversión y Cierre mensual): React + TypeScript + Vite sobre la librería @codeplex-sac, una capacidad por carpeta, navegación data-driven, CRUD por modales y validación de inputs en capas. Cada pantalla habla con el API Go por /api, que ejecuta los procedimientos fn_* en CockroachDB.

Arquitectura

Una capacidad por carpeta, el shell las orquesta

El frontend replica la idea componible del backend: cada capacidad (módulo de negocio) vive en su propia carpeta autocontenida y se registra en un manifiesto. El caparazón (shell) lee qué capacidades están habilitadas y arma la navegación sola; no hay rutas escritas a mano por pantalla.

Regla de oro
El servicio de cada operación es 1:1 con un procedimiento fn_* del backend. La UI nunca arma SQL ni conoce la base: solo llama a su servicio, que invoca el endpoint y traduce la respuesta a { ok, datos, mensajeUsuario }.
Caparazón (shell)
  • DisposicionCaparazon — layout full-width responsive.
  • BarraLateralCaparazon — navegación data-driven.
  • CabeceraCaparazon — hamburguesa móvil.
  • BarraSecciones — pestañas por capacidad.

  • usar_notificacion — toasts.
  • DialogoConfirmacion — confirmar acciones.
  • CodeplexTabla — listados.

Validación de inputs

El componente CampoValidado

Un solo componente envuelve a CodeplexCampoTexto y le agrega contador en vivo, color de borde según el estado y restricción por tipo. El usuario ve cuántos caracteres lleva y si lo que escribe es válido, sin enviar el formulario.

Vacío / sin tocar 0/30
Válido 7/30
Límite alcanzado 30/30
Carácter inválido 3/4

Defensa en capas

1

Frontend

CampoValidado: límite exacto por campo y tipo permitido.

2

API (Go)

Límite de cuerpo 64KB + los tipos numéricos rechazan letras al deserializar.

3

Base de datos

CHECKs de rango y enumeraciones por columna.

Resultado: letras donde van números → bloqueado (modo numérico/decimal + tipos Go). Texto kilométrico → cortado (maxLength en el front, 64KB en el API). Valores fuera de rango → rechazados por los CHECKs de la base.

Conexión con el backend y la base

Del clic al procedimiento, y de vuelta

Toda llamada pasa por invocarAPI. En desarrollo, Vite hace proxy de /api al API Go en el puerto 8080; el API ejecuta el fn_* en CockroachDB y devuelve JSONB que el servicio normaliza.

1

Componente

La página llama a su gancho, que llama al servicio.

2

cliente_api

invocarAPI arma la petición.

3

Proxy /api

Vite → :8080

4

API Go

Endpoint generado → ejecutor → fn_*.

5

CockroachDB

El fn_* valida reglas, escribe y devuelve JSONB.

Multi-tenant: el front nunca manda el id de empresa en el cuerpo: viaja en el header X-Empresa-Id y el backend lo usa para aislar los datos por empresa. El usuario que ejecuta la acción viaja en X-Usuario-Id para la auditoría (creado_por / actualizado_por).

Preguntas frecuentes

¿Por qué un componente CampoValidado y no validar en cada formulario?
Para no repetir lógica: el contador, el color de borde, el maxLength y la restricción por tipo viven en un solo lugar. Cada formulario solo declara modo y límite por campo. Cambiar la regla es cambiar un archivo.
¿La validación del front reemplaza a la del backend?
No, es la primera capa (UX). El backend mantiene el límite de cuerpo y los tipos numéricos, y la base sus CHECKs. Si alguien salta el front, las capas de abajo siguen protegiendo.
¿Cómo sabe la barra lateral qué mostrar?
Lee las capacidades habilitadas del catálogo y pinta un NavLink por cada una. Habilitar o deshabilitar una capacidad es configuración, no código de navegación.
¿Es responsive?
Sí: en escritorio la barra lateral es fija (260px); por debajo de 900px se colapsa en un drawer con botón hamburguesa y fondo oscurecido, el contenido ocupa todo el ancho, los KPIs se apilan y las pestañas de sección se vuelven deslizables.
