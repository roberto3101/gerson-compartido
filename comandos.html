Paso 3

3Generadores (.ps1) — por capacidad

El flujo de Fidel: defines/corriges el SQL → lo declaras en el catálogo → generas. Corre tus 2 capacidades juntas en cada generador.

Catálogos (constantes y descriptores Go/TS)
powershell -ExecutionPolicy Bypass -File .\herramientas\componibilidad\generar_catalogos.ps1
Backend — rutas + tabla de operaciones (los endpoints)
powershell -ExecutionPolicy Bypass -Command "& './herramientas/componibilidad/generar_esqueletos_api.ps1' -codigos_capacidad activos_e_inversion,cierre_mensual"
Frontend — clientes + rutas generadas
powershell -ExecutionPolicy Bypass -Command "& './herramientas/componibilidad/generar_frontend_capacidades.ps1' -codigos_capacidad activos_e_inversion,cierre_mensual"
Leen los archivos .sql de cada capacidad (no el array procedimientos del catálogo), así que generan endpoint para todos los fn_*. Salida: operaciones_generadas_<capacidad>.go + los .ts/.tsx. No se editan a mano.

Paso 4

4Backend (Go) — puerto 8080

servidor Levantar el API (deja la ventana abierta; Ctrl+C para apagar)
cd aplicaciones\api
$env:CADENA_CONEXION_BD = "postgresql://root@localhost:26260/sesga_test?sslmode=disable"

La cadena de conexión NO se commitea. La API la lee de la variable de entorno CADENA_CONEXION_BD (es obligatoria). No hay ninguna cadena en el repo y no debe haberla (lleva credenciales). Solo la seteas aquí, apuntando a tu CockroachDB local y tu base (sesga_test). El código que la lee es base compartida — tampoco se toca.
⚠ Si "No se pudo conectar" o "empresa no vigente": casi siempre es la cadena de conexión, no el código. (1) La API no lee el .env — solo la variable CADENA_CONEXION_BD, y hay que setearla en cada ventana nueva (si quedó de otra sesión apuntando mal, o no la seteas, falla). (2) Apunta al nodo v26 en 26260 y a una base con datos y empresa vigente: sesga_global (trae las semillas demo listas) o sesga_test si armaste esa. (3) Nunca a localhost:26257/defaultdb ni al binario v23 — es un nodo viejo/vacío, la empresa demo no existe ahí y todo falla. Conexión lista para presentar: postgresql://root@localhost:26260/sesga_global?sslmode=disable.
Solo compilar (sin levantar) — para verificar que todo arma

Paso 6

6Probar endpoints

Con el API en :8080. La cabecera X-Empresa-Id es obligatoria (resuelve p_id_empresa); las acciones piden también X-Usuario-Id.

Opción rápida — PowerShell nativo (Invoke-RestMethod)

Definir las cabeceras una vez
$h = @{ "X-Empresa-Id"="11111111-1111-1111-1111-111111111111"; "X-Usuario-Id"="22222222-2222-2222-2222-222222222222" }
GET (lectura) — devuelve 200 con data
Invoke-RestMethod "http://localhost:8080/api/v1/activos_e_inversion/listar_clasificaciones_activo" -Headers $h
POST (comando) — body como objeto, sin archivos ni curl (funciona con el seed mínimo)
$body = @{ id_usuario_accion="22222222-2222-2222-2222-222222222222"; id_tipo_herramienta="a3a3a3a3-a3a3-a3a3-a3a3-a3a3a3a3a3a3"; codigo="HER-DEMO-001"; descripcion="Amoladora demo"; marca="Makita"; modelo="M-100"; numero_serie="SN-DEMO-001"; costo_adquisicion=350; tiempo_vida_meses=24; fecha_inicio_depreciacion="2026-01-01"; fecha_fin_depreciacion="2028-01-01" } | ConvertTo-Json
Invoke-RestMethod "http://localhost:8080/api/v1/activos_e_inversion/registrar_herramienta" -Method Post -Headers $h -ContentType "application/json" -Body $body

Alternativa — curl.exe (no curl a secas en PowerShell)

curl.exe -s -H "X-Empresa-Id: 11111111-1111-1111-1111-111111111111" "http://localhost:8080/api/v1/activos_e_inversion/listar_tipos_herramienta"

Paso 8

8Flujo de ramas

Nunca trabajas directo en capacidad ni en master. Tarea (individual) → capacidad (consolidación) → PR → master. Autor siempre el tuyo.

1 · Crear tu rama de tarea desde la capacidad al día
git switch capacidad/activos-e-inversion
git pull
git switch -c tarea/activos-e-inversion-<tu-nombre>-<entidad>
2 · Commitear con tu autor y subir la tarea
git add .
git -c user.name="<tu-usuario>" -c user.email="<tu-correo>" commit -m "Se ... (mensaje declarativo)"
git push -u origin tarea/activos-e-inversion-<tu-nombre>-<entidad>
3 · Llevar la tarea a la capacidad (fast-forward, NO a master)
git push origin tarea/activos-e-inversion-<tu-nombre>-<entidad>:capacidad/activos-e-inversion
Antes de PR a master: tu rama de capacidad debe estar basada en el codeplexMaster actual. Si está vieja, ponla al día primero — si no, el merge podría revertir el trabajo de otros grupos. Capacidad → master se hace por PR en GitHub (no push directo).

Rama de integración local (local-no-pushear)

Tu vista local de todo integrado (master + tus capacidades) sin pushear. Para refrescarla:

Guardar lo sin commitear, basar en master actual y mergear tus capacidades
git stash -u
git switch local-no-pushear
git reset --hard origin/codeplexMaster
git merge origin/capacidad/activos-e-inversion --no-edit
git merge origin/capacidad/cierre-mensual --no-edit
Regla: snake_case para códigos/rutas (activos_e_inversion, fn_*), kebab para ramas (capacidad/activos-e-inversion). Solo tus archivos, commits declarativos, y local-no-pushear nunca se sube.
