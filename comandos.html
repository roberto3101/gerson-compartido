Comandos para todo

Levantar CockroachDB, construir la base, correr los generadores, levantar backend y frontend, probar endpoints, correr pruebas, y el flujo de ramas — todo en PowerShell, siguiendo la metodología componible del Grupo 5.

PowerShell · Windows CockroachDB v26 · Go · React/Vite
0 · Requisitos 1 · CockroachDB 2 · Base de datos 3 · Generadores 4 · Backend 5 · Frontend 6 · Probar 7 · Pruebas 8 · Flujo git
Ojo con las rutas de tu máquina: la ruta del binario de CockroachDB y la del repo pueden variar. Ajusta los C:\... a donde los tengas. Los puertos: CockroachDB 26260, API 8080, frontend 5173.

Paso 0

0Requisitos

CockroachDB v26.2.1 (el target del proyecto; valida los cuerpos plpgsql al crear). Go, Node/npm. El repo clonado.

# Verificar que tienes lo necesario
go version
node -v ; npm -v
& "C:\cockroachdb26\cockroach-v26.2.1.windows-6.2-amd64\cockroach.exe" version
No uses CockroachDB v23 (no soporta plpgsql en funciones → los CREATE FUNCTION fallan). Usa el binario v26.

Paso 1

1Levantar CockroachDB v26

Una ventana dedicada (déjala abierta; ciérrala para apagar el nodo). Los datos persisten en el --store.

& "C:\cockroachdb26\cockroach-v26.2.1.windows-6.2-amd64\cockroach.exe" start-single-node `
  --insecure --listen-addr=localhost:26260 --http-addr=localhost:8086 `
  --store=C:\cockroachdb26\data
Si sale could not lock temporary directory o address in use, ya hay un nodo corriendo. Ciérralo: Get-Process cockroach | Stop-Process -Force.

Paso 2

2Construir la base de datos

La forma más simple: usar el SQL de descarga del Vercel (trae todas las tablas, índices y procedimientos, y compila en v26). Hace DROP/CREATE de la BD.

Nuestras 2 capacidades → base sesga_test

$CR = "C:\cockroachdb26\cockroach-v26.2.1.windows-6.2-amd64\cockroach.exe"
& $CR sql --insecure --host=localhost:26260 --file sesga_grupo5_capacidades.sql

Todas las capacidades (global) → base sesga_global

& $CR sql --insecure --host=localhost:26260 --file sesga_global_completo.sql

Datos demo (opcional, para que las consultas devuelvan filas)

UUIDs fijos que usa el probador de endpoints. Pégalo a sesga_test:

Paso 3

3Generadores (.ps1) — por capacidad

El flujo de Fidel: defines/corriges el SQL → lo declaras en el catálogo → generas. Corre tus 2 capacidades juntas (si pasas una sola, el agregador deja la otra fuera).

Catálogos (constantes y descriptores Go/TS)

powershell -ExecutionPolicy Bypass -File .\herramientas\componibilidad\generar_catalogos.ps1

Backend (esqueletos API) — rutas + tabla de operaciones

powershell -ExecutionPolicy Bypass -Command "& './herramientas/componibilidad/generar_esqueletos_api.ps1' -codigos_capacidad activos_e_inversion,cierre_mensual"

Frontend (clientes + rutas generadas)

powershell -ExecutionPolicy Bypass -Command "& './herramientas/componibilidad/generar_frontend_capacidades.ps1' -codigos_capacidad activos_e_inversion,cierre_mensual"
Lee los archivos .sql de cada capacidad (no el catálogo), así que genera endpoints para todos los fn_*, estén o no listados en el array procedimientos. Salida: operaciones_generadas_<capacidad>.go y los .ts/.tsx generados.

Paso 4

4Backend (Go) — puerto 8080

Desde la carpeta del API, con la cadena de conexión a CockroachDB. Deja la ventana abierta (es el servidor).

cd aplicaciones\api
$env:CADENA_CONEXION_BD = "postgresql://root@localhost:26260/sesga_test?sslmode=disable"
go build ./...
go run ./cmd/api          # "api sesga reyser escuchando en puerto 8080"
Para apagar: Ctrl+C. Variables opcionales: $env:PUERTO_API="8080", $env:AMBIENTE="desarrollo".

Paso 5

5Frontend (React + Vite) — puerto 5173

cd aplicaciones\web
npm install              # solo la primera vez
npm run dev              # servidor de desarrollo en http://localhost:5173

Build de producción

npm run build            # compila a dist/
El frontend de las capacidades estándar (activos, cierre) se muestra con su página genérica generada. Solo se personaliza a mano si la capacidad necesita una UI especial (como contratos).

Paso 6

6Probar endpoints (curl)

Con el API en :8080. En PowerShell usa curl.exe (no curl a secas). Cabecera obligatoria X-Empresa-Id.

GET (lectura)

$emp = "11111111-1111-1111-1111-111111111111"
curl.exe -s -H "X-Empresa-Id: $emp" "http://localhost:8080/api/v1/activos_e_inversion/listar_clasificaciones_activo"

POST (con cuerpo) — vía archivo, porque PowerShell rompe el JSON inline

'{"id_usuario_accion":"22222222-2222-2222-2222-222222222222","id_contrato":"cccccccc-cccc-cccc-cccc-cccccccccccc","id_periodo":"33333333-3333-3333-3333-333333333333","descripcion":"Penalidad demo","importe":"250.00"}' | Out-File -Encoding ascii body.json
curl.exe -s -X POST -H "X-Empresa-Id: $emp" -H "Content-Type: application/json" -d "@body.json" "http://localhost:8080/api/v1/cierre_mensual/registrar_penalidad"
Más fácil: usa el Probador de endpoints de la metodología — eliges el endpoint, llena los campos y te copia el curl.exe exacto.

Paso 7

7Pruebas (tests SQL)

Las pruebas por capacidad están en el repo. Cada archivo imprime una tabla con PASS/FAIL por caso y un conteo final.

$CR = "C:\cockroachdb26\cockroach-v26.2.1.windows-6.2-amd64\cockroach.exe"
# activos-e-inversion
& $CR sql --insecure --host=localhost:26260 --database=sesga_test --file infraestructura\base-de-datos\pruebas\activos_e_inversion\003_test_integracion.sql
# cierre-mensual (incluye la regla "activo parado no recupera")
& $CR sql --insecure --host=localhost:26260 --database=sesga_test --file infraestructura\base-de-datos\pruebas\cierre_mensual\003_test_integracion.sql
Si fail = 0, todo bien. Corre la BD limpia (paso 2) antes de las pruebas.

Paso 8

8Flujo de ramas (metodología)

Nunca se trabaja directo en la rama de capacidad. Tarea (individual) → merge → capacidad (consolidación) → PR → master.

1) Crear tu rama de tarea desde la capacidad (al día con master)

git switch capacidad/activos-e-inversion
git pull
git switch -c tarea/activos-e-inversion-<tu-nombre>-<entidad>

2) Trabajas, generas y commiteas (con tu autor)

git add .
git -c user.name="<tu-usuario>" -c user.email="<tu-correo>" commit -m "Se ... (mensaje declarativo)"
git push -u origin tarea/activos-e-inversion-<tu-nombre>-<entidad>

3) Merge de tu tarea a la capacidad (NO a master)

git switch capacidad/activos-e-inversion
git merge tarea/activos-e-inversion-<tu-nombre>-<entidad>
git push origin capacidad/activos-e-inversion

4) Capacidad → master: mediante PR en GitHub

Antes de mergear a master: asegúrate de que tu rama de capacidad esté basada en el codeplexMaster actual. Si está vieja, primero ponla al día — si no, el merge podría revertir el trabajo nuevo de otros grupos.
Regla: solo tus archivos, en snake_case para códigos/rutas (gobierno_central, fn_*) y kebab para ramas (capacidad/gobierno-central). Commits declarativos. Nada que mezclar con otras sub-áreas.
