cd aplicaciones\api
$env:CADENA_CONEXION_BD = "postgresql://root@localhost:26260/sesga_test?sslmode=disable"
go build ./...
go run ./cmd/api          # api sesga reyser escuchando en puerto 8080

La idea en una frase

El procedimiento SQL es la única fuente de la verdad

No escribimos un controlador Go por cada operación ni registramos rutas a mano. Definimos la función fn_* en SQL —su nombre y sus parámetros—, la declaramos en el catálogo, y los generadores derivan el endpoint HTTP, el cliente tipado y la ruta del frontend. Si cambia la firma del SQL, regeneramos y todo vuelve a quedar en sincronía.

Regla del proyecto
Dominio SQL → catálogo → generar → personalizar solo si el endpoint necesita lógica que el generador no cubre. El verbo del nombre (listar_, registrar_, ejecutar_…) define el método HTTP; los parámetros p_* definen el contrato de entrada.

El cambio de paradigma

Antes vs. ahora

El método anterior pedía un handler por SP y registro manual de rutas: mucho boilerplate y riesgo de que la firma del SQL y el Go se desincronizaran. El método componible elimina ese trabajo.

● Antes · a mano
  • Un archivo activos_e_inversion.go con una función handler por cada SP (9 funciones casi idénticas).
  • Registrar cada ruta a mano en rutas.go.
  • Repetir en cada handler: validar método, leer cabecera X-Empresa-Id, parsear query, mapear errores.
  • Riesgo de desincronía: si el SQL cambia un parámetro, hay que recordar tocar el Go.
  • Más líneas = más superficie para bugs y para inconsistencias entre capacidades.
● Ahora · generado
  • Cero handlers por SP. Un único ejecutor genérico atiende todas las operaciones.
  • Las rutas se registran solas: el generador emite una tabla de operaciones y el runtime la recorre.
  • Validación de método, inyección de X-Empresa-Id / X-Usuario-Id, coerción de tipos y formato de error: una sola vez, para todos.
  • Imposible desincronizar: la firma del SQL es el contrato; se regenera y queda igual.
  • Frontend incluido: cliente TypeScript tipado + rutas React, también generados.

Las dos fuentes de verdad

Qué leen los generadores

Todo arranca de dos lugares y nada más: el catálogo de capacidades y los archivos SQL de procedimientos.

1 · El catálogo declara qué capacidades existen

capacidades/catalogo_capacidades.json — cada capacidad declara su código, nombre, directorios y segmentos de API/navegación. Es el registro maestro que recorren los tres generadores.

// capacidades/catalogo_capacidades.json  (extracto: nuestras 2 capacidades)
{
  "codigo": "activos_e_inversion",
  "nombre": "Activos e inversion",
  "directorio_datos": "activos_e_inversion",   // dónde viven los .sql
  "directorio_frontend": "activos-e-inversion",
  "segmento_api": "activos_e_inversion",        // → /api/v1/activos_e_inversion/...
  "segmento_navegacion": "activos_e_inversion",
  "procedimientos": [ "fn_listar_activos", "fn_obtener_detalle_activo", ... ]
}

2 · El SQL es el contrato del endpoint

El generador abre cada .sql de la capacidad, detecta CREATE OR REPLACE FUNCTION fn_* y parsea sus parámetros p_* TIPO [DEFAULT]. El nombre define el método y la ruta; los parámetros, el contrato de entrada (con DEFAULT = opcional).

CREATE OR REPLACE FUNCTION fn_listar_activos (
  p_id_empresa      UUID,                 -- requerido → se inyecta de X-Empresa-Id
  p_busqueda        TEXT    DEFAULT NULL,    -- opcional (tiene DEFAULT)
  p_limite          INT     DEFAULT 20,
  p_cursor_creado_en TIMESTAMPTZ DEFAULT NULL,
  p_cursor_id       UUID    DEFAULT NULL
) RETURNS JSONB ...
Por qué funciona: el verbo inicial del nombre (listar_) implica GET; el prefijo p_ se recorta para obtener el nombre de entrada (p_busqueda → busqueda); DEFAULT marca el parámetro como opcional. Toda la información del endpoint ya vive en la firma SQL.

El runtime genérico

Un solo ejecutor para todas las operaciones

La tabla generada no contiene lógica: la atiende operaciones_generadas_base.go, escrito una vez a mano. Recorre las operaciones, registra cada ruta envuelta en su gate de capacidad, y al llegar una petición resuelve los parámetros y llama al SP.

// operaciones_generadas_base.go — registro automático de rutas
func (m *manejadorAPI) registrarRutasGeneradas(mux *http.ServeMux) {
  for _, operacion := range operacionesGeneradas() {   // la tabla generada
    mux.HandleFunc(
      operacion.Patron,                                  // /api/v1/activos_e_inversion/listar_activos
      m.envolverConCapacidad(operacion.CodigoCapacidad,  // gate: ¿la empresa tiene la capacidad activa?
        func(w http.ResponseWriter, r *http.Request) {
          m.ejecutarOperacionGenerada(operacion, w, r)
        }))
  }
}

Qué hace el ejecutor por cada petición

#PasoDetalle
1Valida el métodoSi no coincide con el verbo de la operación → 405 METODO_NO_PERMITIDO.
2Resuelve cada parámetrop_id_empresa se inyecta de la cabecera X-Empresa-Id; p_id_usuario de X-Usuario-Id; el resto, de query string o del cuerpo JSON.
3Verifica obligatoriosSi falta un parámetro sin DEFAULT → 400 PARAMETRO_OBLIGATORIO. Los opcionales ausentes van como NULL.
4Coerciona tiposSegún el tipo SQL: INT→entero, BOOL→booleano, NUMERIC→decimal, resto texto.
5Invoca el SPfunction → EjecutarJSON (devuelve el JSONB crudo); procedimiento → EjecutarProcedimiento (responde un acuse de éxito).
Gratis para todas las capacidades: multi-tenant por cabecera, gate de capacidad, validación de obligatorios, coerción de tipos y formato uniforme de error/respuesta. Nadie reescribe esto por endpoint.

El camino dorado

Flujo para agregar o cambiar una operación

El procedimiento estándar del equipo. Para nuestras dos capacidades este flujo es completo: ninguna de las dos requiere handlers a mano.

1

Escribe el SQL

Crea o ajusta la fn_* en procedimientos/<cap>/. El nombre y los p_* son el contrato.

2

Declara en catálogo

Si es nueva, añádela a catalogo_capacidades.json bajo su capacidad.

3

Genera

Corre los 3 scripts: catalogos → esqueletos_api → frontend.

4

Compila

go build (los scripts ya aplican gofmt). API, cliente y ruta listos.

5

Personaliza solo si

Si el endpoint necesita lógica que el genérico no cubre, recién ahí se escribe a mano.

Orden de versionado del equipo, sin cambios: rama de tarea → se sube a la capacidad → de la capacidad al master componible. El método de generación vive dentro de ese flujo; no lo reemplaza.

La puerta de escape

Cuándo se escribe un handler a mano

La generación es el caso por defecto, no una obligación. Una capacidad puede declararse personalizada y entonces el generador la salta, dejando que conserve sus controladores propios.

# generar_esqueletos_api.ps1
$codigosPersonalizados = @("contratos", "gobierno_central", "produccion")
# → esas 3 conservan handlers a mano (p. ej. contratos.go); el resto se genera.
Se genera (por defecto)
  • activos_e_inversion y cierre_mensual — nuestras capacidades, 100% generadas.
  • costeo_directo, valorización, y toda capacidad CRUD/listado estándar.
  • Cualquier operación que solo reciba parámetros y llame a un SP.
Se escribe a mano (excepción)
  • contratos, gobierno_central, produccion — están en la lista de personalizados.
  • Endpoints con orquestación especial, validación fuera del SP o respuestas no estándar.
  • Se añade el código a la lista y se crea su <cap>.go propio.
Para el Grupo 5: ninguna de nuestras dos capacidades está en la lista de personalizadas. Todo nuestro backend se obtiene de los .sql que ya escribimos, regenerando.

Defensa

Preguntas frecuentes

¿Por qué no escribir los handlers a mano si ya sabemos hacerlo?
Porque cada handler manual repite las mismas 30 líneas (validar método, leer X-Empresa-Id, parsear, mapear error) y abre la puerta a que la firma del SQL y el Go se desincronicen. Con la generación, la firma del SP es el contrato: un solo ejecutor lo cumple para todos y nunca difiere.
Si edito el SQL, ¿qué tengo que tocar en el backend?
Nada a mano. Cambias la fn_*, regeneras (esqueletos_api + frontend) y la tabla de operaciones, el cliente TS y la ruta se actualizan solos. Luego go build para confirmar.
¿Cómo sabe el endpoint a qué empresa pertenece la petición?
El ejecutor detecta el parámetro p_id_empresa y lo inyecta desde la cabecera X-Empresa-Id (multi-tenant). Igual con p_id_usuario desde X-Usuario-Id. El SP siempre recibe la empresa correcta sin que el cliente la mande como parámetro suelto.
¿Qué pasa si una empresa no tiene contratada la capacidad?
Cada ruta generada se envuelve con envolverConCapacidad(codigo, …). Si la empresa no tiene esa capacidad activa, el gate la bloquea antes de ejecutar el SP. Es parte del modelo componible: las capacidades se activan por empresa.
¿Esto reemplaza el flujo de versionado de tarea → capacidad → master?
No. La generación es cómo se produce el código; el versionado es cómo se entrega. Seguimos subiendo rama de tarea, luego a la capacidad y de ahí al master componible. Los archivos generados se versionan como cualquier otro.
